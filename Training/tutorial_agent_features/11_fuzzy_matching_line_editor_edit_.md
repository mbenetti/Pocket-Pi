## The Challenge: Why Simple Search-and-Replace Fails

Imagine an LLM trying to update a function signature. It might offer `def my_function(param1: Any, param2: str):`. However, the actual code on disk could be `def my_function ( param1: Any, param2: str ):` (with extra spaces), or use Windows-style `\r\n` line endings, or even contain smart quotes `‘param1’`. A direct string search for the LLM's suggested code would fail, causing the agent to stumble.

The `edit` tool overcomes these common pitfalls by employing a multi-stage, robust process, inspired by how modern compilers or sophisticated IDEs (like VS Code or JetBrains products) manage code modifications. They don't just rely on raw string manipulation; they often use Abstract Syntax Trees (ASTs) or Document Object Models (DOMs) to identify semantic locations. While Pocket-Pi doesn't build full ASTs, it uses a similar principle of **normalization and target mapping** to achieve comparable precision at the text level.

## The Surgical Pipeline: Precision through Normalization and Mapping

The Fuzzy-Matching Line Editor operates through a carefully orchestrated pipeline that first sanitizes the content, then intelligently locates the desired modification points, and finally applies changes with surgical precision. This process is critical for maintaining the integrity of the target file, much like a version control system ensures merging changes without corrupting the historical record.

Here's the architectural flow:

```mermaid
graph TD
    A[Raw File Content Input] --> B{Determine Original BOM & LF/CRLF};
    B --> C[Normalize Content to LF-only Internally];
    C --> D{Process Each Edit (oldText, newText)};
    D -- Try Exact Match in LF-only String --> E{Did Exact Match Succeed?};
    E -- Yes --> F[Map Char Offset to Line Range];
    E -- No --> G[Apply Fuzzy Normalization (oldText & file content)];
    G --> H[Locate Match in Fuzzy-Normalized String];
    H --> I[Map Fuzzy Char Offset to Original Line Range];
    F & I --> J[Collect Edit Instruction (Start Line, End Line, New Text)];
    J --> K[Sort All Edit Instructions by Line Number (Highest to Lowest)];
    K --> L[Apply Edits to Internal Line Array (Reverse Order)];
    L --> M[Restore Original BOM & LF/CRLF Endings];
    M --> N[Write Modified Content to Disk];
    N --> O[Success/Error Reporting];
```
This multi-step process ensures that:
1.  **Original Formatting is Preserved**: Byte Order Marks (BOM) and line ending styles (`\n` vs `\r\n`) are detected and restored.
2.  **Robust Matching**: Exact matches are prioritized, but fuzzy matching handles common stylistic variations.
3.  **Line-Preserving Transformations**: Modifications are applied in an order that prevents unintended shifts in line numbers, ensuring stable edits.

### 1. Initial Normalization and Line Ending Detection

Before any matching can occur, the file's content is brought to a canonical form, and its original characteristics are recorded for later restoration. This is akin to a compiler first tokenizing and parsing source code into a common intermediate representation before applying transformations.

```python
# from pocket_pi/tools/edit.py
def normalize_to_lf(text: str) -> str:
    """Normalize line endings to LF."""
    return text.replace("\r\n", "\n").replace("\r", "\n")

def detect_line_ending(content: str) -> str:
    """Detect local line ending in a file."""
    crlf_idx = content.find("\r\n")
    lf_idx = content.find("\n")
    if lf_idx == -1: # No newlines at all, assume standard line feed
        return "\n"
    if crlf_idx == -1: # Only LF found
        return "\n"
    return "\r\n" if crlf_idx < lf_idx else "\n" # CRLF if mixed, or if CRLF appears first
```
The `edit_file` function initially reads the `original_content`, detects its `line_ending`, and checks for `has_bom`. Then, `apply_text_edit` receives this `original_content` and immediately converts it to `LF`-only for all internal processing. This establishes a consistent internal representation, much like a database normalizes data to a specific schema to ensure consistency across different input sources.

If an exact match for `oldText` isn't found in the `LF`-normalized content, Pocket-Pi intelligently falls back to a "fuzzy" comparison. ### 3. Absolute Paging Offset Maps: From Characters to Lines

After finding a match (either exact or fuzzy) in the normalized string, the next challenge is to translate those character-based start and end positions back to the original file's _line numbers_. This is vital because the actual modification happens on a line-by-line basis. This technique is analogous to how an operating system maps logical memory addresses used by a program to physical memory addresses on the RAM chips, ensuring memory segments are correctly organized.

```python
# from pocket_pi/tools/edit.py
def get_line_offsets(content: str) -> List[Tuple[int, int]]:
    """Returns a list of (start_char_offset, end_char_offset) for each line in the content."""
    lines = content.split("\n")
    offsets = []
    current = 0
    for line in lines:
        start = current
        end = current + len(line)
        offsets.append((start, end)) # (inclusive start, exclusive end for line content)
        current = end + 1 # +1 for the implicit '\n' character
    return offsets

def find_target_line_range(offsets: List[Tuple[int, int]], start_char: int, end_char: int) -> Tuple[int, int]:
    """Finds which line index range (start_line, end_line) boundary maps to char index boundaries (0-indexed)."""
    start_line = -1
    for idx, (s, e) in enumerate(offsets):
        # Check if the character index falls within this line (including the boundary or line endings)
        if start_char >= s and start_char <= e + 1:
            start_line = idx
            break
            
    if start_line == -1: # Fallback if start_char is before first line
        start_line = 0
        
    end_line = start_line # Initialize end_line to start_line
    for idx in range(start_line, len(offsets)):
        s, e = offsets[idx]
        if s <= end_char: # If this line's start is before/at the target end
            end_line = idx
        else: # If this line's start is beyond the target end, we've found our range
            break
            
    return start_line, end_line
```
*   `get_line_offsets` creates an index map of character boundaries for each line.
*   `find_target_line_range` then uses this map to efficiently translate a character `start_char` and `end_char` into a `start_line` and `end_line` (0-indexed). This gives Pocket-Pi the precise line coordinates required for modification.

### 4. Reverse Order Modifications: The Core of Stability

This is perhaps the most critical design choice for stability. When multiple edits are applied to a file, changing lines at the top of the file can shift the line numbers of all subsequent edits below it. To prevent this, Pocket-Pi applies edits from the bottom of the file upwards. This guarantees that changes to a lower section of the file will not affect the line numbers of any pending modifications in an upper section.

```python
# from pocket_pi/tools/edit.py

        suffix = orig_lines[e_line + 1:]
        
        new_lines = new_text.split("\n") # New text (LF normalized)
        orig_lines = prefix + new_lines + suffix # Construct new line array
```
This strategy is a common pattern in algorithms that modify collections by index, such as deleting elements from a list or applying transformations to a mutable array. By processing from the end, each modification's indices remain valid relative to the remaining elements that have not yet been processed. This mirrors database transaction logging or event sourcing, where operations are often applied in a specific, immutable order to guarantee state consistency.
